Every vector database has two layers: the vectors (heavily optimized, quantized, compressed) and the text payload (UTF-8 at the API, often LZ4-compressed on disk). Nobody talks about compressing that second layer with anything smarter than generic byte codecs.

I ran some benchmarks and found that the payload layer is a real compression opportunity sitting in plain sight. BPE tokenization + a static entropy coder gives **~3× lossless compression** — roughly 2× better than gzip — using infrastructure already in every ML stack. It just needs to be pointed at the right problem.

## The Forgotten Layer

When you insert a document into Qdrant, Weaviate, or Elasticsearch, you typically store something like:

```json
{
  "id": "doc_123",
  "vector": [0.12, -0.34, ...],
  "payload": {
    "text": "Retrieval-Augmented Generation is a technique..."
  }
}
```

The vector gets quantized, compressed, and indexed carefully. The `text` field is still plain UTF-8 at the API — compression with LZ4 happens afterwards.

Qdrant stores on-disk payloads in [Gridstore](https://qdrant.tech/articles/gridstore-key-value-storage/) (its custom KV engine, replacing RocksDB since v1.13) with LZ4 compression enabled by default. Elasticsearch applies LZ4 on stored fields too. Both give you generic byte compression — roughly 2× on text in the benchmarks below — but neither exploits the structure of natural language.

That's the gap.

## How It Works

<TokenCompressionPipeline />

The top path shows what every vector database does today: store text as UTF-8 bytes, then compress with a generic codec. The bottom path is the proposal: tokenize first to get semantically-meaningful chunks, then apply entropy coding on the token IDs.

**Step 1 — BPE tokenization.** BPE (Byte Pair Encoding) merges frequently co-occurring byte sequences into single tokens. " language" becomes one token. " Generation" becomes one token — with the leading space, as the diagram shows. Rarer words split: "Retrieval" is three tokens (`Ret + ri + eval`). On average, one BPE token spans [3/4 of a word](https://help.openai.com/en/articles/4936856-what-are-tokens-and-how-to-count-them). Average English word length is ~5 characters, plus a space = 6 bytes in ASCII/UTF-8:

```
6 bytes/word × 3/4 words/token = 4.5 bytes/token (raw)
2 bytes/token (uint16)
ratio: ~2.25x
```

**Step 2 — ANS entropy coding.** gzip and LZ4 operate on opaque bytes — they can't see that `[0x01, 0x7F]` is token ID 383. ANS (Asymmetric Numeral Systems), the entropy coder inside zstd, can be applied directly on token IDs. It assigns shorter bit codes to frequent tokens: token 198 (`" the"`) is 40× more common than token 47291 (`"embeddings"`), so it gets a much shorter code.

The key design choice: **use a static frequency table trained once on a large corpus**, not per-document frequencies. Per-document ANS looks tempting (it gives even shorter codes by exploiting local repetition), but the decoder needs the same frequency table to reconstruct the document — meaning you'd have to store it alongside each compressed payload. For a ~400-token document that overhead (~900 bytes) completely erases the compression gain. A static table is pre-agreed between encoder and decoder; nothing extra is stored per document.

```python
import constriction, numpy as np, tiktoken

enc   = tiktoken.get_encoding("r50k_base")
VOCAB = 50_257

# ── one-time setup: train on corpus ──────────────────────────────────────────
corpus_ids = enc.encode(corpus_text)
counts = np.bincount(corpus_ids, minlength=VOCAB).astype(np.float64) + 1  # Laplace
probs  = counts / counts.sum()
model  = constriction.stream.model.Categorical(probs, perfect=False)

# ── per-document encode (store this) ─────────────────────────────────────────
def compress(text):
    ids   = enc.encode(text)
    coder = constriction.stream.stack.AnsCoder()
    coder.encode_reverse(np.array(ids, dtype=np.int32), model)
    return len(ids).to_bytes(4, 'big') + coder.get_compressed().tobytes()

# ── per-document decode ───────────────────────────────────────────────────────
def decompress(data):
    n   = int.from_bytes(data[:4], 'big')
    buf = np.frombuffer(data[4:], dtype=np.uint32).copy()
    ids = constriction.stream.stack.AnsCoder(buf).decode(model, n).tolist()
    return enc.decode(ids)
```

The vocabulary size determines the raw packing tradeoff:

```
Tokenizer      Vocab size    ID type    Bytes/token (raw)   + static ANS
─────────────────────────────────────────────────────────────────────────
r50k (GPT-2)    50,257       uint16     2 bytes  ← 2.4x     3.35x ✓
cl100k (GPT-4) 100,277       uint32     4 bytes  ← no win   3.29x ✓
mxbai WP        30,522       uint16     2 bytes  ← 2.4x     ~3.5x*
```

With static ANS, r50k edges out cl100k — cl100k's larger vocabulary means more tokens fall outside the training distribution and get weaker Laplace-smoothed codes.

## The Benchmark

I took a ~270-word paragraph about RAG and vector databases (representative of real payload content) and measured every combination I could think of.

```python
import tiktoken, struct, gzip, zstandard, constriction

enc = tiktoken.get_encoding("r50k_base")
token_ids = enc.encode(text)
packed = struct.pack(f">{len(token_ids)}H", *token_ids)  # uint16 big-endian
```

Here's what I found:

Single reference doc (1,853 bytes, 272 words of technical RAG content):

| Method                       |        Size |      Ratio |     CER |
| ---------------------------- | ----------: | ---------: | ------: |
| Raw UTF-8                    | 1,853 bytes |       1.0x |      0% |
| LZ4 (default in Qdrant/ES)   | 1,468 bytes |   **1.3x** |      0% |
| gzip -9                      |   984 bytes |       1.9x |      0% |
| zstd -22                     |   981 bytes |       1.9x |      0% |
| brotli q=11                  |   711 bytes |       2.6x |      0% |
| r50k BPE uint16 (raw)        |   784 bytes |   **2.4x** |      0% |
| r50k BPE + zstd              |   709 bytes |   **2.6x** |      0% |
| zstd dict 112KB (WikiText)   |   747 bytes |       2.5x |      0% |
| **r50k BPE + static ANS**    |   648 bytes | **2.9x** ✓ |      0% |
| mxbai WordPiece + static ANS |  ~630 bytes |    **~3x** | 20.1%\* |
| Static entropy lower bound   |   651 bytes |      ~2.8x |       — |

\*mxbai CER drops to 0% after simple post-processing — more on this below.

Averaged over 1,773 WikiText-103 test articles (in-distribution):

| Method                     | Avg ratio |
| -------------------------- | --------: |
| gzip -9                    |     1.66x |
| zstd -22                   |     1.68x |
| brotli q=11                |     2.24x |
| zstd dict 112KB (WikiText) |     2.58x |
| **r50k + static ANS**      | **3.35x** |

zstd with a 112KB corpus-trained dictionary reaches 2.58× — a strong result, but still 30% behind static ANS. Brotli's built-in web dictionary gets to 2.24× with no training at all. The remaining advantage of tokenization comes from operating on semantic units: `" embeddings"` is a single token with a known global frequency; byte n-gram models have to learn that from scratch on every corpus.

The headline: **raw BPE uint16 already beats every byte codec on raw text, with zero further compression** — including LZ4, which is what Qdrant and Elasticsearch actually use (1.3×). That's just tokenization and packing. Adding a static ANS coder trained on WikiText-103 pushes the single-doc number to 2.9× and the corpus average to **3.35×** — more than 2× better than LZ4 with no domain-specific tuning.

### Latency (2,000 runs, 1,853-byte doc)

Compression ratio is only half the story. Full pipeline numbers:

| Pipeline                              |   Median |     p99 |
| ------------------------------------- | -------: | ------: |
| r50k encode (tokenize + ANS)          |    156µs |   287µs |
| r50k decode (ANS + detokenize)        | **41µs** |    53µs |
| cl100k encode (tokenize + ANS)        |    145µs |   224µs |
| cl100k decode (ANS + detokenize)      | **39µs** |    55µs |
| mxbai encode (tokenize + ANS)         |    750µs | 1,270µs |
| mxbai decode (ANS + detokenize + fix) |    420µs |   603µs |

- **The tokenizer dominates encode, not ANS.** r50k tokenization is 132µs; ANS adds only 30µs. If you already tokenize at ingest, the marginal encode cost is those 30µs.
- **Reads are fast.** r50k decode takes 41µs total — negligible on any serving path.
- **ANS is symmetric** — encode and decode are nearly identical (~31µs each), unlike zstd where decompress is 45× faster than compress.
- **cl100k is faster than r50k** (145µs vs 156µs encode) despite uint32 IDs — fewer tokens per document more than offset any overhead. ANS latency is indifferent to token ID width.

## The mxbai Story: Better Compression, Surprising Catch

<details>
<summary>mxbai WordPiece gives **5.7× compression** — 10% better than r50k — but is 5–10× slower and lowercases text. Expand for details.</summary>

The mixedbread embedding model (`mxbai-embed-large-v1`) uses BERT's WordPiece tokenizer with 30,522 tokens. Smaller vocabulary → more common tokens → lower entropy → better compression.

mxbai + static ANS reaches roughly **630 bytes** vs r50k + static ANS at **648 bytes** — a small compression win. But when I measured latency and recovery, the story got more complicated.

| Tokenizer       |   CER | CER (ignore case+whitespace) | CER (+ fix punct) |
| --------------- | ----: | ---------------------------: | ----------------: |
| r50k BPE        |  0.0% |                         0.0% |              0.0% |
| mxbai WordPiece | 20.1% |                         6.4% |          **0.0%** |

The 20% CER is entirely from two deterministic artifacts of BERT tokenization:

1. **Lowercasing** — `"Qdrant"` → `"qdrant"`, `"RAG"` → `"rag"`
2. **Punctuation spacing** — `"(content)"` → `"( content )"`, `"up-to-date"` → `"up - to - date"`

Neither is information loss at the token level. The information is there — it's just encoded with different whitespace conventions. Three regex rules fix it completely:

```python
def fix_punct(s):
    s = re.sub(r'\( ', '(', s)
    s = re.sub(r' \)', ')', s)
    s = re.sub(r'(\w) - (\w)', r'\1-\2', s)
    return s
```

So mxbai is effectively lossless too, with a small detokenizer — but it's also **5× slower to encode** (750µs vs 156µs) and **10× slower to decode** (420µs vs 41µs), entirely due to HuggingFace tokenizer overhead. The marginal compression win doesn't justify it. For a storage codec, r50k is the clean choice.

On UNK tokens: WordPiece has a `[UNK]` token that causes **actual** information loss. But for standard English text — including code, URLs, numbers, gibberish ASCII — it never fires. It only fires on emojis (🚀 → `[UNK]`) and some rare Unicode math symbols. So for typical RAG payloads over web or document text, UNK is a non-issue.

</details>

## Real-World Scale: English Wikipedia

English Wikipedia is ~7M articles, 708 words/article on average — a useful benchmark for RAG-scale corpora.

Extrapolating from the corpus-average 3.35× (WikiText-103 test split, 1,773 articles):

| Method                |         Per doc |    7M docs |    vs raw |
| --------------------- | --------------: | ---------: | --------: |
| Raw UTF-8             |     4,497 bytes |    31.5 GB |        1x |
| zstd -22              |     2,382 bytes |    16.7 GB |      1.9x |
| r50k + zstd           |     1,721 bytes |    12.1 GB |      2.6x |
| **r50k + static ANS** | **1,342 bytes** | **9.4 GB** | **3.35x** |

At 1B docs the storage gap is meaningful:

| Method            |    1B docs | S3 ($0.02/GB/mo) | Disk ($0.08/GB/mo) |
| ----------------- | ---------: | ---------------: | -----------------: |
| Raw UTF-8         |     4.5 TB |             ~$92 |              ~$368 |
| r50k + static ANS |     1.3 TB |             ~$27 |              ~$108 |
| **Saving**        | **3.2 TB** |      **~$65/mo** |       **~$260/mo** |

Disk savings are the bigger line item (~$260/month on SSD at 1B docs), and the I/O win is proportional: 1.3 TB reads 3.4× faster than 4.5 TB, directly reducing query latency when payloads are fetched during retrieval. S3 egress compounds it — at $0.09/GB, shipping 3.2 TB fewer data saves ~$290 per bulk transfer.

## How This Compares to State of the Art

For context, here's where this sits on the broader compression landscape (benchmarked on enwik9, 1GB Wikipedia):

| Method                                   | bits/byte | Ratio vs raw | Practical?     |
| ---------------------------------------- | --------: | -----------: | -------------- |
| Raw UTF-8                                |       8.0 |           1x | —              |
| gzip -9                                  |      ~2.6 |         3.1x | Yes            |
| zstd -22                                 |      ~2.0 |           4x | Yes            |
| **BPE + static ANS (this post)**         |  **~2.4** |    **~3.3x** | **Yes**        |
| zpaq                                     |      ~1.4 |         5.7x | Slow           |
| fx2-cmix (Hutter Prize record, Oct 2024) |     0.887 |           9x | Extremely slow |
| Chinchilla 70B (DeepMind, 2024)          |     0.664 |          12x | Not practical  |

The gap between r50k + static ANS (~3.3x) and the Hutter Prize record (~9x) is exactly the signal that lives in **conditional token probabilities**. Static ANS only models marginal frequency — how often each token appears globally. The top compressors model `P(token | all previous tokens)`, which is what LLMs do. That's the remaining 3x on the table.

Getting from 3x to 9x requires a language model. Getting from 1x to 3x just requires a tokenizer and 30 lines of Python.

## What Needs to Change

Nothing in the current stack prevents this. The pieces exist:

- `tiktoken` is already a dependency in any LLM-adjacent stack
- `constriction` is a Rust-backed library with a Python API, used in ML compression research
- Decoding is fast — ANS is symmetric, O(n) in both directions

What would need to change in a vector DB like Qdrant:

1. Accept `encoding: "r50k"` as a payload storage option
2. Pack token IDs as uint16, run ANS on the stream
3. Decompress transparently on read

The index, vectors, and query pipeline stay identical. This is purely a payload storage optimization.

<!-- DIAGRAM: side-by-side of current architecture vs proposed architecture showing payload storage layer -->

```
Current                          Proposed
───────────────────────────────────────────────────────
INSERT                           INSERT
  payload: {"text": "..."}         payload: {"text": "..."}
       │                                  │
       ▼                                  ▼
  JSON → Gridstore (LZ4)         tokenize → uint16 IDs → ANS
                                        │
                                        ▼
                                  Gridstore (ANS)

READ                             READ
  fetch LZ4 JSON bytes             fetch ANS-compressed bytes
       │                                  │
       ▼                                  ▼
  return as-is                   ANS decode → token IDs → decode
                                        │
                                        ▼
                                  return original text
```

The entire round-trip is transparent to the user. Same API, ~3x smaller payloads.

As [search and generation converge](https://www.kshivendu.dev/blog/why-search#the-duality-of-search:~:text=Unified%20search%20and%20generation), keeping UTF-8 blobs in your index while the model reasons in tokens means maintaining two representations of the same text. Token-native payload storage collapses that to one.

## Key Takeaways

- Vector databases optimize vectors aggressively; text payloads get generic LZ4 at best (**1.3×** — that's the actual Qdrant/Elasticsearch default, not gzip). Token-aware compression is a separate opportunity on top.
- BPE tokenization (r50k, vocab=50,257) produces uint16 token IDs — 2 bytes each — that already beat gzip on raw text with zero further compression (2.4× vs 1.9×).
- Adding a **static ANS entropy coder** trained once on WikiText-103 brings the average to **3.35×** — about 2× better than gzip. The static table is the right design: per-document ANS requires storing a ~900-byte frequency table per doc to be decodable, which completely erases the compression gain on typical RAG chunk sizes.
- r50k edges out cl100k for static ANS (3.35× vs 3.29×) because cl100k's larger vocabulary leaves more tokens under-represented in any fixed corpus table.
- r50k is perfectly lossless (0% CER). mxbai WordPiece achieves marginally better compression but lowercases text — fixable with post-processing — and is 5–10× slower to tokenize.
- **Latency is not a concern:** the tokenizer dominates encode (~132µs); ANS with a pre-loaded static model adds only ~30µs. Decode is ~40µs total.
- The gap between this (~3.3×) and the Hutter Prize record (~9×) is conditional token probability — modelling `P(token | context)` instead of just global marginals. Getting from 3× to 9× requires a language model; getting from 1× to 3× just requires a tokenizer and 30 lines of Python.
- At English Wikipedia scale (7M docs), this takes payload storage from 31.5 GB to ~9.4 GB. The bigger win is I/O: 3.4× faster bulk reads, indexing, and replication.
