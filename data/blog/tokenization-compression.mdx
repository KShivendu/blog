Every vector search engine (aka vector DB) stores two things for each point: the vector and the payload. Vectors get all the attention: quantization, compression, careful index design. The text in the payload gets LZ4 at best.

This post makes one claim: **tokenize that text before compressing it, and you roughly double what generic byte codecs achieve**. I'll show the benchmark, the latency cost (small), and what it would take to add this to a real database.

There's another reason to care than the compression ratio: agents, not humans, are increasingly the ones reading unstructured data, and agents read and write in tokens. A **token-native storage format** for text speaks their language directly and gets the compression win for free.

## The Forgotten Layer

When you insert a document into Qdrant, or Elasticsearch, you typically store something like:

```json
{
  "id": 123,
  "vector": [0.12, -0.34, ...],
  "payload": {
    "text": "Retrieval-Augmented Generation is a technique..."
  }
}
```

Qdrant stores on-disk payloads in [Gridstore](https://qdrant.tech/articles/gridstore-key-value-storage/) and compresses each payload value independently with LZ4, so a point read never has to decompress a neighboring document, keeping random lookups cheap. Elasticsearch instead packs stored fields into 16 KB blocks and compresses each block together, letting LZ4 back-reference across documents at the cost of decompressing the whole block to read one field. Either way, the codec has no learned model of the language, just whatever byte patterns happen to repeat within its window. That gets you roughly 1.15x per-document (Qdrant) to 1.85x block-level (Elasticsearch-style) on text.

But there is a component that knows a lot about the structure of natural language, and it's already part of every embedding and language model: **the tokenizer**.

## How It Works

<TokenCompressionPipeline />

The top path is what databases do today: store UTF-8 bytes, compress with a generic codec. The bottom path is my proposal: tokenize first, then entropy-code the token IDs.

**Step 1: BPE tokenization.** BPE (Byte Pair Encoding) starts with raw bytes and repeatedly merges the most frequent adjacent pair into a new token, building up a vocabulary of common subwords. OpenAI ships two BPE tokenizers: **r50k** (used by GPT-2/GPT-3, 50,257 tokens) and **cl100k** (used by GPT-3.5/GPT-4, 100,277 tokens). cl100k has a larger vocabulary so it learns longer merges, which usually (but not always) means fewer tokens per document, especially for code and non-English text. Either way, each token ID needs 4 bytes (uint32) since the vocab exceeds the uint16 limit of 65,535. r50k fits in 2 bytes (uint16).

BPE merges frequent byte sequences into single tokens. `"language"` is one token. `"Retrieval"` is three (`Ret + ri + eval`). On average one BPE token covers [3/4 of a word](https://help.openai.com/en/articles/4936856-what-are-tokens-and-how-to-count-them). An average English word is ~5 characters plus a space, so:

```
6 bytes/word × 3/4 words/token = 4.5 bytes/token (raw text)
2 bytes/token (r50k uint16 token IDs)
ratio: ~2.25x
```

With this napkin math, we see that just packing token IDs into uint16 already compresses, before any actual compression algorithm runs. This is the napkin math that started my whole investigation: if packing alone gets ~2.25x for free, an actual entropy coder on top of it should do even better, which is what the rest of this post sets out to measure.

**Step 2: ANS entropy coding.** gzip and LZ4 see opaque bytes and can't know that `[0x01, 0x7F]` is token 383. An entropy coder working directly on token IDs can give frequent tokens short codes: `"the"` is 40x more common than `"embeddings"`, so it gets a much shorter code. I used ANS (Asymmetric Numeral Systems, the entropy coder inside zstd) via the [constriction](https://github.com/bamler-lab/constriction) library.

One design choice matters here: the frequency table is **trained once on a large corpus and shared**, not built per document. A per-document table compresses better on paper, but the decoder needs the table to reconstruct the document, so you'd have to store it alongside each payload. That's ~900 bytes for a typical 400-token chunk, which erases the entire gain. A shared static table costs nothing per document. It's completely possible to learn this static table on a general corpus and reuse it everywhere.

<details>
<summary>The whole thing is ~30 lines of Python. Expand to see.</summary>

```python
import constriction, numpy as np, tiktoken

enc   = tiktoken.get_encoding("r50k_base")
VOCAB = 50_257

# ── one-time setup: train on corpus ──────────────────────────────────────────
corpus_ids = enc.encode(corpus_text)
counts = np.bincount(corpus_ids, minlength=VOCAB).astype(np.float64) + 1  # Laplace
probs  = counts / counts.sum()
model  = constriction.stream.model.Categorical(probs, perfect=False)

# ── per-document encode (store this) ─────────────────────────────────────────
def compress(text):
    ids   = enc.encode(text)
    coder = constriction.stream.stack.AnsCoder()
    coder.encode_reverse(np.array(ids, dtype=np.int32), model)
    return len(ids).to_bytes(4, 'big') + coder.get_compressed().tobytes()

# ── per-document decode ───────────────────────────────────────────────────────
def decompress(data):
    n   = int.from_bytes(data[:4], 'big')
    buf = np.frombuffer(data[4:], dtype=np.uint32).copy()
    ids = constriction.stream.stack.AnsCoder(buf).decode(model, n).tolist()
    return enc.decode(ids)
```

</details>

## The Benchmark

Measured across 1,773 [WikiText-103](https://huggingface.co/datasets/Salesforce/wikitext) test articles (≥30 words). The ANS frequency table was trained on the WikiText-103 train split, a shared static model with no per-document overhead.

| Method                             |       min |    median |       max |
| ---------------------------------- | --------: | --------: | --------: |
| LZ4, per-document (Qdrant)         |     0.88x |     1.15x |     1.78x |
| LZ4, 16KB blocks (ES-style)        |     1.35x |     1.85x |     2.33x |
| gzip -9                            |     0.99x |     1.66x |     2.34x |
| zstd -22                           |     1.11x |     1.68x |     2.39x |
| brotli q=11                        |     1.22x |     2.26x |     3.13x |
| r50k BPE uint16 (raw)              |     1.04x |     2.33x |     3.34x |
| cl100k BPE uint32 (raw)            |     0.45x |     1.13x |     1.62x |
| zstd trained dict 112KB (WikiText) |     1.51x |     2.61x |     3.63x |
| **r50k BPE + static ANS**          | **1.68x** | **3.37x** | **4.30x** |
| **cl100k BPE + static ANS**        | **1.68x** | **3.30x** | **4.14x** |

All methods above are lossless (0% character error rate on full round trip).

The single-document LZ4 row is what a naive per-document codec gets. The 16KB-block row is the fairer real-world number: Elasticsearch packs ~20 documents per block and compresses them together, so LZ4 gets to back-reference across documents, not just within one.

<details>
<summary>Method glossary: what the names mean</summary>

- **LZ4**: default codec in Qdrant and Elasticsearch. Optimized for speed, not ratio.
- **gzip -9**: max compression level, classic deflate. The `-9` is the level (1–9).
- **zstd -22**: [Zstandard](https://github.com/facebook/zstd) at level 22 ("ultra" mode). Much slower to compress than default, same decompression speed.
- **brotli q=11**: Google's codec at max quality. Best ratio of the byte codecs, but slow.
- **r50k BPE uint16 (raw)**: r50k has 50,257 tokens, which fits in uint16 (max 65,535), so 2 bytes per token ID.
- **cl100k BPE uint32 (raw)**: cl100k has 100,277 tokens, which exceeds uint16 and requires uint32 (4 bytes per token). It produces fewer tokens per document than r50k (larger vocab = longer merges), but 4 bytes × fewer tokens still comes out worse than 2 bytes × more tokens. Hence 1.13x vs 2.33x.
- **zstd trained dict 112KB**: zstd with a 112KB dictionary trained on WikiText-103 (`zstd --train`), a global dictionary shared across all documents, same design as the static ANS table. The fairest byte-codec comparison since it also learns from the corpus.
- **r50k / cl100k BPE + static ANS**: ANS allocates bits based on token frequency, not a fixed byte width. A common token gets ~1 bit and a rare one gets ~16 bits, regardless of vocab size. So the uint32 penalty that crushes cl100k raw disappears entirely, and both tokenizers land near 3.3x.

</details>

Two things stand out. r50k raw uint16 (no compression algorithm at all) already beats every byte codec except brotli. And r50k + static ANS beats everything by a wide margin. cl100k raw uint32 sits near 1.13x, worse than even gzip, yet cl100k + ANS recovers to 3.30x: the fixed-width storage penalty vanishes once ANS allocates bits by frequency instead of by vocab size.

The fairest competitor is zstd with a corpus-trained dictionary (`zstd --train`), since it also learns from the corpus. It reaches 2.61x median, still 30% behind. The difference is that the token model operates on units of language: `"embeddings"` is one symbol with a known global frequency, where a byte-level model has to rediscover that pattern from scratch.

### What does it cost in latency?

Measured across 200 WikiText-103 articles (50–500 words), static corpus ANS model:

| Method       | enc median | enc p99 | dec median | dec p99 |
| ------------ | ---------: | ------: | ---------: | ------: |
| LZ4          |       15µs |    36µs |        1µs |     2µs |
| gzip -9      |       23µs |    44µs |       19µs |    30µs |
| zstd -22     |       68µs |   193µs |        3µs |     6µs |
| zstd dict    |      129µs |   326µs |        2µs |     5µs |
| r50k + ANS   |       54µs |   129µs |       18µs |    44µs |
| cl100k + ANS |       55µs |   156µs |       18µs |    51µs |

LZ4 decompresses in 1µs, designed entirely for speed. zstd dict's encoder is slowest at 129µs (best of three dictionary sizes tested: 32KB, 64KB, 112KB), though it decompresses in 2µs. The tokenizer methods sit comfortably in the middle: ~54µs encode (dominated by the tokenizer, not ANS), ~18µs decode.

Note that today's embedding models use different tokenizers (WordPiece, SentencePiece) than LLM BPE tokenizers, so you can't skip the tokenization step by reusing embedding tokenization. The real benefit runs the other way: if you store token IDs in LLM format (r50k/cl100k), you avoid re-tokenizing retrieved documents at inference time when feeding them to the LLM, since the token IDs are already there.

<details>
<summary>Side quest: mxbai's WordPiece tokenizer compresses slightly better but lowercases your text. Expand if curious.</summary>

The mixedbread embedding model (`mxbai-embed-large-v1`) uses BERT's WordPiece tokenizer with a 30,522-token vocabulary. Smaller vocabulary means more common tokens, lower entropy, slightly better compression. Across 1,773 WikiText-103 test articles:

| Method                       |   min | median |   max |
| ---------------------------- | ----: | -----: | ----: |
| r50k BPE uint16 (raw)        | 1.04x |  2.33x | 3.34x |
| mxbai WordPiece uint16 (raw) | 0.88x |  2.35x | 3.34x |
| r50k BPE + static ANS        | 1.68x |  3.37x | 4.30x |
| mxbai WordPiece + static ANS | 1.63x |  3.56x | 4.54x |

Raw uint16 packing is basically a wash between the two (2.35x vs 2.33x): mxbai's smaller vocabulary barely matters before any entropy coding runs. The gap only opens up once ANS is applied, ~5% better compression at the median. The problem is **character error rate (CER)**: 80.4% on average, every single article affected. Punctuation-spacing errors (regex-fixable, `( x )` → `(x)`) are a minor, acceptable cost, but they aren't the bulk of it: fixing them only brings CER down to 81.3%, statistically the same. The real damage is BERT's lowercasing: `"Qdrant"` → `"qdrant"`, `"Wikipedia"` → `"wikipedia"`. WikiText is dense with capitalized words (titles, proper nouns, sentence starts), so virtually every article is corrupted, and case, unlike punctuation spacing, isn't recoverable after the fact. mxbai isn't a viable _lossless_ codec for real text, though if lowercase-everywhere is acceptable for your use case, the compression is genuinely better.

</details>

## What It Saves at Scale

English Wikipedia is ~7M articles averaging 708 words. Applying the corpus-average ratios:

| Method                |         Per doc |    7M docs |    vs raw |
| --------------------- | --------------: | ---------: | --------: |
| Raw UTF-8             |     4,497 bytes |    31.5 GB |        1x |
| zstd -22              |     2,382 bytes |    16.7 GB |      1.9x |
| **r50k + static ANS** | **1,342 bytes** | **9.4 GB** | **3.35x** |

At a billion documents that's 1.3 TB instead of 4.5 TB. At \$0.08/GB for SSD and \$0.02/GB for S3-class object storage:

| Storage tier   | Raw UTF-8 | r50k + ANS |   Saved |
| -------------- | --------: | ---------: | ------: |
| SSD ($0.08/GB) |   $360/mo |    $107/mo | $253/mo |
| S3 ($0.02/GB)  |    $90/mo |     $27/mo |  $63/mo |

Every snapshot, replica, and bulk transfer shrinks by the same 3.35× factor. The I/O win is often worth more than the storage bill: 3.35x fewer bytes means 3.35x faster bulk reads, indexing, and replication.

One scoping note so the claim doesn't overreach: in collections with large embeddings, the vectors dominate total bytes (a 1536-dim float32 vector is ~6 KB, often bigger than the text next to it). Payload compression matters most when payloads are large, vectors are quantized, or text is the point of the collection.

## How Far Can This Go?

The static frequency table only knows how often each token appears globally. The next step up is a bigram table (`P(token | previous token)`) which costs 1.4 MB and gets you to about 4.35x. A full bigram table (69 MB) reaches ~4.7x.

<details>
<summary>How can a probabilistic model recover the exact text?</summary>

The probability model only controls how many bits each token gets. The token IDs themselves are always stored and recovered exactly. Nothing is approximated.

Take the sequence `" language" → " model" → " retrieval"`:

- **Unigram ANS**: `" model"` appears in 0.3% of all tokens → ~8 bits
- **Bigram ANS**: `" model"` after `" language"` is very common → maybe ~4 bits

The decoder recovers `" model"` exactly either way. It just decoded `" language"` first, so it knows to use `P(· | " language")` as the ruler for the next token, and gets the exact same ID back.

Think of it as Morse code with a smarter codebook. E gets one dot not because "E" is approximate, but because it's frequent. A better frequency estimate means shorter codes, not lossy data.

</details>

Past that is LM territory. Chinchilla 70B achieves 0.664 bits/byte on Wikipedia (roughly 12× compression) because a language model is literally computing `P(token | all previous tokens)`, which is the optimal entropy coder by definition. The gap between 3.3× and 12× is entirely conditional probability.

| Method                           | Ratio     | Practical?            |
| -------------------------------- | --------- | --------------------- |
| **BPE + static ANS (this post)** | **~3.3×** | **Yes**               |
| BPE + bigram table               | ~4.5×     | Yes (1.4 MB overhead) |
| Chinchilla 70B                   | ~12×      | No                    |

And because the stored format is just token IDs, swapping the unigram table for a bigram table later is a codec upgrade, not a data migration.

## What Would a Database Need to Do?

Nothing exotic. Any store that compresses text payloads on write and decompresses on read can slot this in as an alternative codec:

1. Accept a token-native encoding (e.g. `encoding: "r50k"`) as a payload storage option
2. Tokenize on insert, pack IDs as uint16, run ANS with the shared static table
3. Decompress transparently on read

The index, vectors, and query pipeline don't change at all, since this only touches how the payload bytes are stored and retrieved.

```
Current                          Proposed
───────────────────────────────────────────────────────
INSERT                           INSERT
  payload: {"text": "..."}         payload: {"text": "..."}
       │                                  │
       ▼                                  ▼
  JSON → generic codec (LZ4)     tokenize → uint16 IDs → ANS
                                        │
                                        ▼
                                  store (ANS)

READ                             READ
  fetch compressed JSON bytes      fetch ANS-compressed bytes
       │                                  │
       ▼                                  ▼
  return as-is                   ANS decode → token IDs → decode
                                        │
                                        ▼
                                  return original text
```

There is one real cost: the vocabulary and frequency table become part of the on-disk format forever, so encodings have to be versioned per collection.

There's also a bigger-picture reason to like token-native storage as [search and generation converge](https://www.kshivendu.dev/blog/why-search#the-duality-of-search:~:text=Unified%20search%20and%20generation): keeping UTF-8 blobs in the index while the model reasons in tokens means maintaining two representations of the same text.

The same trick applies to **agent trace storage**. Tool calls, reasoning steps, and retrieved context pile up as unstructured text, and the logs are typically compressed with the same generic byte codecs. Token-native encoding gives you the same ~3× win there, and the stored token IDs are what the agent would produce anyway when replaying or analyzing a run.

Push this further and the interesting question isn't just how to store tokens, it's whether text needs to come back at all. If the consumer reading a stored document is an LLM, not a human, decoding ANS-compressed bytes back to token IDs is already the end of the job: the next step is feeding those IDs into a model, which tokenizes internally anyway. The UTF-8 round trip only exists to serve human eyes. In a pipeline you fully control, retrieval through generation on a model you host yourself, token IDs never have to become text at all except for logs or a UI. That only works end to end if you own the inference stack, since hosted LLM APIs take text prompts, not raw token arrays, for safety and injection-guarding reasons, and it means standardizing on one tokenizer's ID space rather than juggling several. But it's the logical endpoint of treating tokens as the native format instead of a compression trick bolted onto text.

I'd also expect more overlap in LLM tokenizers over time (cl100k is already a de facto baseline across several model families) and common standards should emerge. That makes token-native storage more natural: you're not locking into one vendor's vocabulary so much as aligning with where the ecosystem is heading.

## A Note to Tokenizer Vendors

Every tokenizer vendor already computes a token frequency table as a byproduct of training BPE merges on their corpus. Publishing it, versioned alongside each tokenizer release (`r50k.freq`, `cl100k.freq`, `o200k.freq`), costs them nothing: no new training, no risk of leaking raw documents, just aggregate counts they already have sitting in a file.

Token pricing and inference latency stay untouched, since billing and speed depend on inference-time token count, not how those IDs get stored downstream. Context window is also unaffected, since that's a function of vocabulary design, not frequency-table publishing. What it actually removes is the bootstrapping tax every downstream team pays today: this post had to approximate GPT's real distribution using WikiText, an English-only proxy, when a real corpus-derived table would need no approximation at all.

The one place a real table wins clearly is multilingual and code coverage. GPT's actual training mix already includes code and dozens of languages, so a vendor-published table would generalize where a WikiText proxy can't. Our JSON benchmark showed exactly this failure mode: a text-trained table gave 1.10× on JSON, a JSON-trained table gave 2.24×. A vendor's real table, trained on a mix that already includes structured and multilingual text, would likely land close to that JSON-trained number out of the box, for every content type at once.

It's the same category of thing as a font shipping kerning tables or JPEG shipping default quantization tables: cheap metadata release, no product cost, and it unlocks compression work the vendor has no reason to do themselves.

## Limitations

- **Decode is slower than LZ4, the codec this post is actually trying to replace.** zstd isn't deployed in Qdrant or Elasticsearch; LZ4 is, chosen precisely because it's fast, not because it compresses well. Against LZ4, ANS decode is 18µs versus LZ4's 1µs, an 18x gap, and encode is 54µs versus 15µs. Both are still tiny next to typical network or disk I/O, so it's a non-issue in absolute terms, but relatively speaking LZ4 wins on speed by a wide margin, which is exactly the tradeoff it was built for. Even the fairest byte-codec competitor, zstd with a trained dictionary, decodes faster than ANS (2µs vs 18µs); it just loses on ratio and encode speed.
- **The frequency table is only as good as the corpus it's trained on.** This post approximates GPT's real token distribution using WikiText-103, an English-only proxy. Domain mismatch is expensive: a WikiText-trained table gets 1.10x on JSON, a JSON-trained table gets 2.24x. Anyone deploying this needs a table trained on their actual data, not a borrowed one.
- **Sharing only works at the corpus level, not per block.** LZ4 gets a real, free win from sharing a compression window across a block of documents. The same trick backfires for ANS: a table trained fresh on each 16KB block costs 900 to 4,000 bytes of overhead just to ship the table, which loses to the static corpus-wide table even at 1MB blocks.
- **cl100k's larger vocabulary doesn't always mean fewer tokens.** On long digit strings and emoji-heavy text it can produce more tokens than r50k, since its extra vocabulary budget went toward code and non-English scripts, not those cases.
- **The vocabulary and frequency table become part of the on-disk format forever.** Change either and every previously encoded document needs re-encoding or a versioned codec path.

## Key Takeaways

- Vector databases apply LZ4 to text payloads (1.15x). Tokenizing first and entropy-coding the token IDs gives 3.35x on average. Lossless, 2.9x better than LZ4.
- Raw uint16 token IDs alone reach 2.4x, beating every byte codec except brotli, with no compression algorithm at all.
- The frequency table must be static and shared. Per-document tables cost ~900 bytes each and erase the gain.
- Latency is a non-issue: ANS adds ~30µs to encoding (the tokenizer you may already run dominates), and decode is ~41µs.
- The same idea stretches to ~4.7x with a bigram table (1.4 MB of table already gets 4.35x). Past ~5x you need an actual language model.
- The trade-off to respect: the vocabulary becomes part of your on-disk format forever, so version your encodings.
