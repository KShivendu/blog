BM25 is the workhorse of lexical search. It's fast, requires no GPU, and is surprisingly competitive on most retrieval benchmarks. Its weakness is well-known: it can only match exact terms. A query for "cardiac arrest" won't retrieve a document that says "heart attack", even if that document is the best answer.

SPLADE fixes this by using a neural network to expand documents into a richer sparse representation — one that activates "cardiac" when it sees "heart attack." The catch is query latency: running a transformer at search time adds 50–100ms per query.

**Inference-free SPLADE** removes that catch entirely. The model runs only at index time on documents. Queries use raw BERT token IDs — no model, no GPU, no neural inference. You pay once upfront; queries stay fast.

Here's how it compares across six retrieval setups, with a result that surprised me.

## How It Works

Standard SPLADE encodes both sides neurally:

```
query  → SPLADE encoder → sparse vector
doc    → SPLADE encoder → sparse vector
                        → dot product → score
```

Inference-free SPLADE (SPLADE-IF) encodes only documents:

```
query  → BERT tokenizer → raw token IDs (weight=1.0)
doc    → SPLADE encoder → sparse vector
                        → dot product → score
```

The document vectors still carry neural expansions — "heart" activates "cardiac", "coronary", "myocardial". The query side is just tokenization: "cardiac arrest" becomes token IDs `[3684, 6295]` each with weight 1.0. No model call, no GPU.

```python
def encode_query_inference_free(tokenizer, query: str):
    enc = tokenizer(query, add_special_tokens=False, truncation=True, max_length=512)
    unique_ids = list(set(enc["input_ids"]))
    return SparseVector(indices=unique_ids, values=[1.0] * len(unique_ids))
```

At query time: one tokenizer call + one sparse dot product. That's it.

## The Experiment

I benchmarked six configurations on [BEIR scifact](https://github.com/beir-cellar/beir) — 5,183 documents, 300 test queries, scientific claim verification:

| Method               | Doc encoding                    | Query encoding                | Backend       |
| -------------------- | ------------------------------- | ----------------------------- | ------------- |
| BM25-Qdrant          | `Qdrant/bm25` fastembed         | fastembed BM25                | Qdrant sparse |
| BM25-pyserini        | Lucene BM25                     | Lucene BM25                   | Lucene        |
| SPLADE-IF-Qdrant     | `naver/splade-v3-doc` (offline) | raw BERT tokens               | Qdrant sparse |
| SPLADE-IF-pyserini   | `naver/splade-v3-doc` (offline) | raw BERT tokens               | Lucene BM25   |
| SPLADE-Full-Qdrant   | `Splade_PP_en_v1` (offline)     | `Splade_PP_en_v1` (per query) | Qdrant sparse |
| SPLADE-Full-pyserini | `Splade_PP_en_v1` (offline)     | `Splade_PP_en_v1` (per query) | Lucene BM25   |

SPLADE doc encoding ran on a Modal A10G GPU (~3 min for 5K docs). Lucene-based methods convert SPLADE float weights to repeated token strings for indexing — more on why this matters below.

| SPLADE-IF-pyserini   | 0.6344     | 0.5894     | 0.7566     | 0.6026     | 6.6ms / 10.2ms / 13.4ms        | 0.2ms     | 6.4ms     |

Three findings stand out.

## Finding 1: SPLADE-IF Nearly Matches Full SPLADE at 15× Lower Latency

SPLADE-Full-Qdrant and SPLADE-IF-Qdrant are nearly indistinguishable on quality:

|                    | NDCG@10 | MAP@10  | MRR@10  | Recall@10  |
| ------------------ | ------- | ------- | ------- | ---------- |
| SPLADE-Full-Qdrant | 0.7093  | 0.6664  | 0.6811  | 0.8240     |
| SPLADE-IF-Qdrant   | 0.7068  | 0.6644  | 0.6748  | **0.8254** |
| Δ                  | −0.0025 | −0.0020 | −0.0063 | +0.0014    |

The quality difference is noise-level. But the latency difference is not: full SPLADE costs **62.5ms median** per query vs **4.1ms** for SPLADE-IF — a 15× gap, nearly all of it in the 55ms query encoder call.

The reason inference-free works so well: `naver/splade-v3-doc` is an **asymmetric model** trained specifically for this regime. Its documents are encoded to be maximally useful when matched against unweighted query tokens. It bakes query-side expansion into document representations at index time, so raw token matching is surprisingly effective.

The sparsity numbers confirm this:

|             | Doc avg non-zeros | Query avg non-zeros   |
| ----------- | ----------------- | --------------------- |
| SPLADE-IF   | 325               | 20 (raw tokens)       |
| SPLADE-Full | 205               | 65 (neural expansion) |

SPLADE-IF documents are denser (325 nnz vs 205) — they carry richer expansions to compensate for sparse queries. SPLADE-Full spreads the work across both sides. Both approaches converge to similar quality.

Both methods beat BM25-Qdrant meaningfully: **+3.5% NDCG@10, +2.1% Recall@10, +4.3% MRR@10** for SPLADE-IF, at essentially the same 4ms query latency.

## Finding 2: The Retrieval Backend Matters Enormously for SPLADE

The most striking result is the performance gap between Qdrant and Lucene when SPLADE vectors are involved:

| Method      | Qdrant     | Lucene/pyserini | Gap        |
| ----------- | ---------- | --------------- | ---------- |
| BM25        | 0.6830     | 0.6789          | −0.6%      |
| SPLADE-IF   | **0.7068** | 0.6344          | **−10.2%** |
| SPLADE-Full | **0.7093** | 0.6542          | **−7.8%**  |

BM25 is essentially unchanged across backends. SPLADE loses 8–10% NDCG@10 when moved to Lucene. Why?

SPLADE assigns **continuous float weights** to each token. Qdrant stores and scores these directly. Lucene is a text search engine — to use SPLADE expansions, you must convert float weights to something Lucene understands:

```python
def splade_to_text(indices, values, tokenizer) -> str:
    # weight 3.7 → repeat token 4 times
    # weight 0.4 → include token once
    tokens = tokenizer.convert_ids_to_tokens(indices)
    parts = [token] * max(1, min(20, round(weight))
             for token, weight in zip(tokens, values))
    return " ".join(parts)
```

This discretization loses precision. A token with weight 3.7 and one with weight 2.1 both end up repeated 4 times and 2 times respectively — the ratio is preserved roughly, but the absolute signal is mangled. Lucene's BM25 then applies its own IDF and TF normalization on top of these counts, compounding the distortion.

Qdrant's native sparse scoring computes the exact dot product over float weights. The information is preserved end-to-end.

This has an important implication: **benchmarks that evaluate SPLADE over Lucene indices are likely underestimating the method's true quality.** The Qdrant backend is the right substrate for SPLADE vectors.

## Finding 3: Why SPLADE-IF and BM25 Have the Same Query Latency

The "inference-free" name can mislead — it sounds like SPLADE-IF is doing less work than BM25. In practice both reduce to the same operation: a **sparse dot product** in Qdrant.

| Step                | BM25-Qdrant                     | SPLADE-IF-Qdrant   |
| ------------------- | ------------------------------- | ------------------ |
| Query prep          | fastembed tokenize + IDF lookup | BERT tokenize only |
| Qdrant call         | sparse dot product              | sparse dot product |
| Query embed latency | 0.1ms                           | 0.3ms              |
| Search latency      | 4.0ms                           | 3.8ms              |

The search latency (the Qdrant dot product) is actually slightly _faster_ for SPLADE-IF (3.8ms vs 4.0ms). The sparse vectors are different shapes — SPLADE-IF query vectors have ~20 non-zeros (raw tokens) while BM25 query vectors have all document terms weighted by IDF — but Qdrant handles both efficiently.

The tiny embed difference (0.1ms BM25 vs 0.3ms SPLADE-IF) comes from fastembed's BM25 also doing IDF weighting; BERT tokenization alone is slightly more work than a vocabulary lookup, but both are sub-millisecond.

## The Index-Time Cost

The tradeoff is entirely upfront. Encoding 5,183 documents with `naver/splade-v3-doc`:

- **CPU**: ~45+ minutes (no GPU available locally)
- **Modal A10G GPU**: ~3 minutes (batch_size=32 to avoid OOM on 22GB VRAM)

For a static or slowly-updated corpus this is very acceptable. For a high-churn index you'd need an async pipeline that re-encodes new documents as they arrive — the same problem dense vector search already solves with embedding pipelines.

Once indexed, queries cost nothing extra. This is the asymmetry SPLADE-IF exploits: **encode once, query forever.**

## Context: Qdrant's Own Research

Qdrant's [miniCOIL paper](https://qdrant.tech/articles/minicoil) argues that SPLADE _without term expansion_ performs worse than BM25. Their [e-commerce fine-tuning series](https://qdrant.tech/articles/sparse-embeddings-ecommerce-part-1) similarly found a 6× quality drop when replacing the query encoder with static embeddings on Amazon product data.

Our scifact results tell a different story. The discrepancy likely comes down to two factors:

1. **Model choice**: `naver/splade-v3-doc` is purpose-built for asymmetric (inference-free) use. The models used in those posts were trained symmetrically and not designed to compensate for raw query tokens.

2. **Domain**: Scientific claim verification relies heavily on vocabulary bridging ("genome sequencing" ↔ "DNA analysis"). A strong doc encoder can fully anticipate these expansions upfront. E-commerce search has different characteristics — product titles are already fairly exact-match, so document-side expansion helps less.

The inference-free approach works best when the domain demands vocabulary generalization and you pick an asymmetric model trained for it.

## Key Takeaways

- **SPLADE-IF nearly matches full SPLADE quality** — the difference is 0.0025 NDCG@10 in favor of full SPLADE, while query latency is **15× faster** (4.1ms vs 62.5ms). The asymmetric `naver/splade-v3-doc` model is specifically trained to make this work.
- **Both beat BM25** by +3.5% NDCG@10, +4.3% MRR@10 at the same ~4ms latency. The improvement is real and consistent.
- **Backend selection matters for SPLADE far more than for BM25.** Qdrant's native sparse float scoring outperforms Lucene by 8–10% NDCG@10 for SPLADE, vs <1% for BM25. Float weight precision is lost in the text-repetition conversion Lucene requires.
- **The embed/search split reveals where latency lives.** For full SPLADE, 55ms of the 62ms median is query encoding. The Qdrant search itself (6.9ms) is comparable to BM25. Inference-free eliminates that 55ms entirely.
- **Query vector sparsity** is 20 non-zeros for SPLADE-IF (raw tokens) vs 65 for full SPLADE (neural expansion), yet both achieve the same search latency. SPLADE-IF doc vectors are denser (325 vs 205 nnz) — the model pushes expansion responsibility to index time.
- **Index cost is the only real tradeoff**: ~3 min on an A10G GPU for 5K docs. Acceptable for static corpora; requires an async re-encoding pipeline for high-churn indexes.
