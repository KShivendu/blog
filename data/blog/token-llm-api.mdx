[The token-native storage post](/blog/token-storage) argued for storing text as BPE token IDs instead of UTF-8 bytes: embedders, rerankers, and LLMs all consume tokens, not bytes. But I flagged one limit there, reading a payload back as token IDs only pays off end to end "if you own the inference stack, since hosted LLM APIs take text prompts, not raw token arrays (yet)." This post is about that "yet": what would it take for a hosted LLM API to accept token IDs as input and return them as output, instead of forcing a detokenize-then-retokenize round trip on every call?

## What Hosted APIs Do Today

Every hosted chat API is text in, text out, even though the model samples token IDs at every step. The token-ID plumbing already exists one layer down, in the serving stacks, it just isn't exposed:

| Surface                             | Input                       | Output          | Token IDs?            |
| ----------------------------------- | --------------------------- | --------------- | --------------------- |
| OpenAI Chat Completions / Responses | text / content blocks       | text (+`usage`) | ✗                     |
| Anthropic Messages                  | text / content blocks       | text            | ✗ (no `logprobs` )    |
| OpenAI legacy Completions           | text **or array of number** | text            | ✅ input, frozen 2023 |
| vLLM (`skip_tokenizer_init`)        | `prompt_token_ids`          | token IDs       | ✅ both (server-side) |
| TGI (`details`)                     | text                        | per-token `id`  | ✅ output             |

- **The one hosted precedent is stranded.** OpenAI's legacy Completions `prompt` field has always typed as "string, array of string, array of number, or array of array of number", and "array of number" is literally a list of token IDs from the model's own tokenizer. Documented, shipped behavior, not a proposal. But it only works with legacy completion models (`gpt-3.5-turbo-instruct` and older), and OpenAI froze the endpoint in mid-2023. The precedent exists in the same company's API surface, just parked on a model family nobody points new traffic at.
- **The pattern already runs server-side.** vLLM's `LLM` class takes `prompt_token_ids` directly, and `skip_tokenizer_init` goes further: with it set, the engine expects token IDs in and returns token IDs out, no text touches the request or response path. TGI's `details` mode returns a per-token integer `id` on output (its input routes still take text). So the proposal below is a hosted wrapper around something already in production, not new serving tech.

Output is text on every hosted chat endpoint, with token _counts_ in `usage` but never the token _ID sequence_ the model sampled. The closest thing is `logprobs`/`top_logprobs`, and it looks token-adjacent but isn't:

```json
{
  "token": " model",
  "bytes": [32, 109, 111, 100, 101, 108],
  "logprob": -0.234,
  "top_logprobs": [
    { "token": " model", "bytes": [32, 109, 111, 100, 101, 108], "logprob": -0.234 },
    { "token": " system", "bytes": [32, 115, 121, 115, 116, 101, 109], "logprob": -2.891 }
  ]
}
```

That carries the decoded token _string_, its raw UTF-8 _bytes_, and a _log probability_, but never the integer vocabulary ID. I can reconstruct the text (the API already does that) and rank candidates by probability, but I can't recover which row of the embedding table `" model"` was without re-tokenizing the string myself and hoping my tokenizer matches the server's exactly. Anthropic's Messages API doesn't expose `logprobs` at all; there's a community feature request, not a shipped feature.

## A Concrete Interface Proposal

The [storage post](/blog/token-storage) solved almost this exact interface problem once, for the read and write paths of a vector DB payload. The same shape carries over with a swap of nouns: prompt for payload, model tokenizer for collection tokenizer.

**Input.** A chat message's `content` already accepts a string (text) or an array of content parts (for multimodal messages). The simplest token-ID form reuses the storage post's exact string-or-array trick, dispatched on shape: a string is text, a bare array of integers is token IDs. No new block type needed, and it mirrors OpenAI's legacy Completions `prompt`, which already unions string with array-of-number.

```js
// Current: content is text
POST /v1/chat/completions
{
  "model": "gpt-5.1",
  "messages": [
    { "role": "user", "content": "Summarize the attached document." }
  ]
}
```

```js
// Proposed: content is a bare array of token IDs, told apart from text by shape
POST /v1/chat/completions
{
  "model": "gpt-5.1",
  "tokenizer_version": "gpt-5.1-2026-07",
  "messages": [
    { "role": "user", "content": [3838, 42891, 262, 5301, 6190, 13] }
  ]
}
```

The one case a bare array can't cover is interleaving token IDs with other modalities in the same message (token IDs, then an image). That's exactly what the content-parts array is for, so wrap them in a part alongside the others only when you need that:

```js
"content": [
  { "type": "input_tokens", "token_ids": [3838, 42891, 262, 5301, 6190, 13] },
  { "type": "image_url", "image_url": { "url": "..." } }
]
```

**Output.** Mirror the storage post's read side, where `with_payload` grew a per-field format string (`"text"` vs `"tokens"`) instead of staying a boolean:

```js
POST /v1/chat/completions
{
  "model": "gpt-5.1",
  "tokenizer_version": "gpt-5.1-2026-07",
  "response_format_tokens": true,
  "messages": [ ... ]
}
```

```json
{
  "choices": [
    {
      "message": {
        "role": "assistant",
        "content": "The document proposes...",
        "token_ids": [464, 6190, 26057, 986]
      }
    }
  ]
}
```

Returning both `content` and `token_ids` costs nothing extra, the server has the ID sequence before it decodes to text, decode is the cheap direction. A caller that wants tokens only, to skip shipping text at all, asks for `token_ids` and omits `content`, exactly like the storage read path asking for `"tokens"` only.

**Streaming** carries the same field per chunk instead of only `delta.content`:

```json
{ "choices": [{ "delta": { "content": " model", "token_ids": [2746] } }] }
```

The one thing storage didn't need is `tokenizer_version`. A collection's tokenizer is a durable, collection-level choice; a hosted API's tokenizer changes across model versions, so the request has to pin the exact tokenizer it tokenized against, not just the model family.

## Why It Has To Be Opt-In and Versioned

- **Token IDs are only meaningful against the exact vocabulary that produced them.** Pin the wrong tokenizer version and the integers decode to garbage, a worse failure than today's text interface, where a prompt string is portable across every model forever because text has no version.
- **Non-portable across model families.** r50k and cl100k IDs mean nothing to a Claude or Llama tokenizer, so sending token IDs commits you to one provider's one tokenizer version for that request, not a provider-agnostic prompt.
- **So you always need a text fallback.** [The tokenizer-translation post](/blog/tokenizer-translation) showed decode-then-re-encode across tokenizers is lossless and cheap (**121.6µs** median) precisely because it routes through text as the portable intermediate. A token-ID API can't take that shortcut for a cross-provider fallback, a degraded route, or a migration, it still has to go through text on both sides.

## The Payoff

For a system already storing tokens natively, this removes the last detokenize/retokenize pair in the pipeline. Today, even with token-native storage, every LLM call forces two conversions that change nothing about the data: the retrieved payload gets detokenized to text to build the prompt, and the model's output gets detokenized to text before it reaches you, only for you to retokenize it to store it token-natively again.

The storage post measured tokenization at **~450µs per read** of a 512-token chunk. A token-ID request skips it on prompt-build and skips it again on output-store: retrieve token IDs, send them as the prompt, receive token IDs back, store them. Text materializes only at the edges, where a human actually needs to read something.

## Limitations

- **This is a proposal, not a shipped feature.** No major hosted chat API accepts token-ID input or returns token-ID output. Everything under the interface proposal is my design.
- **The precedent is narrower than it reads.** OpenAI's legacy Completions `prompt` field really does accept "array of number" today, but it's frozen to legacy models since 2023. vLLM's `prompt_token_ids`/`skip_tokenizer_init` and TGI's per-token `id` are real and in production, but at the serving-stack layer, not behind any hosted API.
- **Anthropic's `logprobs` situation is unclear** from public sources beyond a community request; if they ship it, the "what APIs do today" comparison may already be stale.
- **Versioning is a real adoption blocker.** A provider shipping this has to keep old tokenizer versions addressable indefinitely, or every token-ID request breaks the moment they retrain a tokenizer, a burden text never imposes.
- **Providers may have good reasons not to.** Text is where content-safety filtering, prompt-injection detection, and moderation run; a raw token-ID path is a second entry point that bypasses whatever of that is string-pattern-based. Token IDs are also an obfuscation channel, a sequence decoding to something a filter would catch can be built directly in ID space. And I don't know how token-ID input interacts with prefix-based prompt caching. I don't have rebuttals for these, worth testing.

## Key Takeaways

- Hosted chat APIs are text in, text out today, OpenAI Chat/Responses and Anthropic Messages both.
- OpenAI's legacy Completions API is real precedent for token-ID input, its `prompt` field accepts arrays of token IDs, but only on legacy models frozen since 2023.
- vLLM proves token-ID input and output works in production, just below the hosted-API line, not on it.
- The interface reuses the storage post's design: a type-detected block for input, a format flag for output, plus a `tokenizer_version` pin, and it can't be portable across model families the way text is.
- The payoff is for systems already storing tokens: retrieve, prompt, receive, and store as token IDs, with text materializing only at the human-facing edges.
