Latency vs Recall

We'll study:

- Effect of increasing dataset size: `range(0, 10M, 100M, 1B)`
- Effect of increasing dims

We'll do it with the following:

- Effect of binary quantization

And we'll do this with two different approaches:

- Brute force vector comparisons (with SIMD optimizations)
- HNSW

All numbers below are measured with [`bfb`](https://github.com/qdrant/bfb) against an **isolated** Qdrant node (16-core / 64 GB, ARM; vectors in RAM), server-side p50, 8 concurrent queries (`-p 8`), on random vectors. Single run — treat as preliminary.

The dashed line is a pure `O(N)` fit to the clean 100k–1M region (**a = 3.7×10⁻⁴ ms/vector, R² = 0.9996**) — full scan is textbook-linear there. But I'm holding off on strong conclusions, because the large-N fp32 points fall off that line:

- The fp32 curve is **not monotonic** — 5M (1849 ms) comes out _slower_ than 10M (1197 ms), and this reproduces across runs even though the node, RAM residency, and segment count (8) are identical. The tails are heavy too (fp32 p99 is 4–5× p50).
- This is **not** noisy neighbours — the node is dedicated and nothing else is querying it. The likely cause is the measurement setup: 8 concurrent queries (`-p 8`), each fanning across 8 segments on 16 cores, is heavily oversubscribed, so per-query latency gets sensitive to scheduling. Binary quant is far steadier here since each scan touches ~32× less data.

The next step is to control for that — repeat each point, sweep concurrency, and likely pin the scan to a fixed parallelism so latency reflects data size rather than the scheduler.

fp32 full-scan latency is **linear in dimension** — the dashed fit `0.94·dim + 129 ms` lands R² = 0.9998. High-dimensional embeddings are exactly where binary quant pays off most.

### Time complexity analysis of HNSW vector indexing:

Each HNSW insertion needs a search, and assuming it takes K ops to add the found K links. This means insertion time is:

(O(log1) + k) + (O(log2) + k) + .... (O(logN) + k)

Sigma(log(I), i*from=1, i_to=N) + kN
= log(1 * 2 \_ 3 \* .... N) + kN
= log(N!) + kN

We have Stirling's approximation for large N:

log(N!)≈NlogN−N

log(N!)+kN≈NlogN−N+kN=NlogN+(k−1)N

Since K is constant, time complexity is O(NLogN) which is very bad

### HNSW in practice

That `O(N log N)` build cost isn't just theory — here's what it looks like on the same node. Each collection was uploaded with `indexing_threshold = 0` (no index), then flipped to `1` to trigger a full HNSW build, timing the indexing, then searched.

The dashed lines are a one-parameter fit to `t = a·N·log₂(N)` — the exact `O(N log N)` the analysis above predicts. It nails the data: **a = 4.3×10⁻⁵ s for fp32 (R² = 0.999)** and **a = 1.3×10⁻⁵ s for BQ (R² = 0.9996)**. Adding a linear `+b·N` term barely moves R², so a single constant is enough — the build really is N log N, no fudge factors. fp32's constant is ~3.2× BQ's: full-precision distance during graph construction is that much costlier than 1-bit-per-dim codes.

fp32 build climbs from 50 s (100k) to ~80 min (5M); the **10M fp32 build blew past a 2-hour cap and is being re-measured** (this chart will get its 10M fp32 point). Binary quant builds **3–6× faster** at every size.
