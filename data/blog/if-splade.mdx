BM25 is the workhorse of lexical search. It's fast, requires no GPU, and is surprisingly competitive on most retrieval benchmarks. Its weakness is well-known: it can only match exact terms. A query for "cardiac arrest" won't retrieve a document that says "heart attack", even if that document is the best answer.

SPLADE fixes this by using a neural network to expand documents into a richer sparse representation. Here's what [`naver/splade-v3-doc`](https://huggingface.co/naver/splade-v3-doc) actually activates for "heart attack":

The original tokens score highest, but semantically related terms — `cardiac`, `stroke`, `disease`, `chest`, `death` — are all activated with meaningful weights. A query for "cardiac arrest" hits `cardiac` (weight 0.77) even though the document only mentions "heart attack." BM25 would miss it entirely.

The catch is query latency: running a transformer at search time adds 50–100ms per query.

**Inference-free SPLADE** removes that catch entirely. The model runs only at index time on documents. Queries use raw BERT token IDs — no model, no GPU, no neural inference. You pay once upfront; queries stay fast.

Here's how it compares across six retrieval setups, with two results that surprised me.

## How It Works

<SpladeVsIF />

The document vectors still carry neural expansions — "heart" activates "cardiac", "coronary", "myocardial". The query side is just tokenization: "cardiac arrest" becomes token IDs `[3684, 6295]` each with weight 1.0. No model call, no GPU.

```python
def encode_query_inference_free(tokenizer, query: str):
    enc = tokenizer(query, add_special_tokens=False, truncation=True, max_length=512)
    unique_ids = list(set(enc["input_ids"]))
    return SparseVector(indices=unique_ids, values=[1.0] * len(unique_ids))
```

At query time: one tokenizer call + one sparse dot product. That's it.

## Results

I benchmarked a full matrix of configurations on [BEIR scifact](https://github.com/beir-cellar/beir) (5,183 docs, 300 queries, scientific claim verification): three SPLADE models × two backends × full and inference-free mode, plus BM25 on both backends. SPLADE doc encoding ran on a Modal A10G GPU. [`naver/splade-v3`](https://huggingface.co/naver/splade-v3) and [`naver/splade-v3-doc`](https://huggingface.co/naver/splade-v3-doc) are the same model family — symmetric vs asymmetric training — giving the cleanest apples-to-apples measure of dropping query-side inference. The third model, [`opensearch-neural-sparse-encoding-doc-v3-gte`](https://huggingface.co/opensearch-project/opensearch-neural-sparse-encoding-doc-v3-gte), is an asymmetric doc-only encoder from OpenSearch.

## Finding 1: SPLADE-IF Nearly Matches Full SPLADE at 13× Lower Latency

The cleanest comparison is within the same model family — [`naver/splade-v3`](https://huggingface.co/naver/splade-v3) (full) vs [`naver/splade-v3-doc`](https://huggingface.co/naver/splade-v3-doc) (inference-free). Dropping query-side inference costs only **−0.0093 NDCG@10** (under 1.3% relative). The latency difference is not noise: full SPLADE costs **57ms median** per query vs **4.3ms** for SPLADE-IF — a 13× gap driven entirely by the 50ms query encoder call.

The reason inference-free works so well: [`naver/splade-v3-doc`](https://huggingface.co/naver/splade-v3-doc) is an **asymmetric model** trained specifically for this regime. Its documents are encoded to be maximally useful when matched against unweighted query tokens. It bakes query-side expansion into document representations at index time, so raw token matching is surprisingly effective.

The sparsity numbers confirm this — SPLADE-IF doc vectors are denser (314 avg non-zeros) than the naver full model (288 nnz), carrying richer expansions to compensate for sparse queries. SPLADE-Full spreads the work across both sides; both approaches converge to similar quality.

Both methods beat BM25-Qdrant meaningfully: **+3.5% NDCG@10, +2.1% Recall@10, +4.3% MRR@10** for SPLADE-IF, at essentially the same 4ms query latency.

## The Tradeoff: Free Queries, Expensive Index

At query time, SPLADE-IF and BM25 are the same operation — a **sparse dot product** in Qdrant. Both take ~4ms. The "inference-free" name can mislead; there's no neural model removed from the query path relative to BM25, just relative to full SPLADE.

The cost is entirely upfront. SPLADE-IF models encode at 58–83 docs/sec on an A10G GPU vs BM25's 1,439 docs/sec on CPU — about 17–25× slower. At that rate, a million-document corpus takes 3–5 hours. For a static or slowly-updated corpus this is a one-time cost. For high-churn indexes you'd need an async re-encoding pipeline — the same problem dense vector search already solves.

Once indexed: **encode once, query forever.**

## Limitations

**Asymmetric training matters.** The approach works because `naver/splade-v3-doc` was trained specifically for raw-token queries. We also tested inference-free against [`prithivida/Splade_PP_en_v1`](https://huggingface.co/prithivida/Splade_PP_en_v1) (a symmetric model): NDCG@10 dropped to 0.6859 on Qdrant and 0.6704 on pyserini — still above BM25 on both backends, but well below the asymmetric result. The ~2% gap between symmetric and asymmetric IF holds consistently across both backends. If you already have a symmetric SPLADE index, dropping the query encoder still gives a meaningful free speedup with a moderate quality tradeoff. For best results, use an asymmetric model.

**No query-time adaptation.** Standard SPLADE can expand "heart attack" → cardiac, coronary, myocardial at query time. Inference-free can't — the doc encoder must anticipate every useful expansion upfront. On truly novel terms (new drug names, recent product launches, breaking-news proper nouns), SPLADE-IF degrades toward BM25 behavior. Full SPLADE handles this better via query-time expansion; dense models handle it differently via subword composition, though dense retrieval trades this for semantic drift on rare terms.

**Domain sensitivity.** Inference-free works best when vocabulary bridging is the core challenge and the domain vocabulary is predictable — scientific, medical, legal. On e-commerce, Qdrant's own experiments found a 6× quality drop because product titles are already close to exact-match with query terms. If your queries are short and already literal ("black 42mm apple watch"), document-side expansion helps less.

**Denser index and GPU required at index time.** SPLADE-IF doc vectors average 314 non-zeros vs 205 for full SPLADE — more memory and disk. BM25 indexes at 1,439 docs/sec on CPU; SPLADE-IF needs an A10G at 83 docs/sec. Not a query concern, but a real operational cost for high-churn corpora.

## Key Takeaways

- **SPLADE-IF nearly matches full SPLADE quality** — within the same model family, dropping query-side inference costs only 0.0093 NDCG@10 (under 1.3% relative) while cutting query latency **13×** (57ms → 4.3ms). Both beat BM25 by +3.5% NDCG@10 at essentially the same latency.
- **Asymmetric training is the key lever for IF quality.** Using a symmetric model (PP) in inference-free mode costs ~2% NDCG@10 vs the dedicated asymmetric model ([naver/splade-v3-doc](https://huggingface.co/naver/splade-v3-doc)). If you have an existing symmetric index, IF still beats BM25, but use an asymmetric model for best results.
- **The only real tradeoff is index time.** Queries are free once indexed. The GPU cost at index time (83 docs/sec on A10G) is acceptable for static corpora; high-churn indexes need an async re-encoding pipeline.
